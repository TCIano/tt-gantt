# APS 高级排产甘特图组件检测文档（自研版）

> 用途：把本文件连同你的组件信息（代码片段/目录结构/截图/说明）一起发给任意 AI，让 AI 按此清单输出“是否符合 APS 高级排产甘特”的结论与差距清单。  
> 适用对象：Vue/React/原生前端均可；尤其适合“纯自研甘特组件”做架构与能力体检。

## 0. 你需要提供给 AI 的材料（Evidence Pack）

为避免 AI 猜测，请尽量提供以下证据（至少提供 A 类 + B 类各 1 项）：

### A 类：结构证据（必选其一）
- A1. 项目目录树（`tree -L 4` 输出或截图）
- A2. 组件核心文件列表（含路径）+ 每个文件一句话职责

### B 类：关键实现证据（建议都给）
- B1. 数据模型定义：`Task/Resource/Link/Calendar/Scenario`（类型/接口/类）
- B2. 布局计算：时间→x、资源行→y、任务条布局、依赖线 path 的代码
- B3. 交互：拖拽移动/拉伸/连线创建的核心处理逻辑
- B4. 状态管理：store（Pinia/Vuex）或内部状态设计
- B5. Undo/Redo：命令栈或撤销实现
- B6. 校验与规则：资源日历校验、依赖硬约束、冲突检测等

### C 类：运行效果证据（可选但很有帮助）
- C1. 录屏/动图：拖拽、拉伸、连线编辑、冲突提示、缩放
- C2. 截图：资源树、左侧表格冻结列、甘特区、依赖线、不可用时段

> 建议：把证据按上述编号命名，AI 引用时更清晰（例如“见 B3”）。

## 1. 检测目标与判定口径

### 1.1 APS 甘特 vs 项目管理甘特
APS（高级排产）甘特图的核心是 **“资源约束排程 + 工序序列编辑”**，而不是项目计划展示。  
因此本检测优先关注：
- 资源/日历/约束/冲突/锁定/插单/批量调度
- 可视化编辑闭环（改得动、改得稳、可回滚、可对比方案）

### 1.2 判定输出（AI 必须给出）
AI 评审输出必须包含：
1) **结论等级**（见第 6 节评分）  
2) **最关键的 3 个缺口**（按优先级排序）  
3) **推荐的改造路线**（“不推倒重来”的最小重构路径）  
4) 若不符合 APS：明确说明它更像哪类组件（展示型甘特 / 项目甘特 / 时间轴列表等）

## 2. APS 业务能力检测清单（“能不能用于排产”）

> 说明：每条打分 0/1/2：  
> 0=没有；1=部分/勉强；2=完整且可用。并要求 AI 引用证据编号（如“B3”）。

### 2.1 数据与对象模型（必须项）
1. 资源轴（Y）支持资源树（工厂→车间→产线→设备/工作中心），可展开折叠（证据：B1/C2）  
2. 任务条（X）为工序/工单/批次：至少包含 `resourceId + start/end`（证据：B1）  
3. 日历/班次/不可用时段：支持显示与校验（证据：B1/B6/C2）  
4. 依赖关系 Link：支持 FS/SS/FF/SF + lag（证据：B1/B2/C2）  
5. 锁定/冻结：任务可锁、最好支持范围锁定（证据：B1/B3/B6）  
6. 状态/告警：缺料/超期/冲突/超载等能以颜色/角标表达（证据：C2/B1）

### 2.2 浏览与对齐（基础可用性）
7. 多粒度缩放：分钟/小时/天/周（证据：C1/C2/B2）  
8. 时间轴可配置：班次边界、节假日（证据：B2/B6/C2）  
9. 左侧表格 + 冻结列（工单字段等）与甘特联动（证据：C2）  
10. 行过滤/搜索/只看异常（证据：C2 或说明）

### 2.3 高级交互（APS 核心）
11. 拖拽移动任务（跨资源行/同资源内）并实时校验（证据：B3/C1）  
12. 拉伸任务时长并校验（证据：B3/C1）  
13. 插单：未排任务拖入资源行并吸附空隙（证据：C1/B3）  
14. 冲突可视化：重叠、不可用占用、依赖违反（证据：B6/C1/C2）  
15. 依赖连线绘制与编辑（创建/删除/改类型/改 lag）（证据：B3/C1）  
16. 批量操作：多选移动/一键顺延/批量锁定（证据：B3/C1）  

### 2.4 资源负载（你们的关键需求之一）
17. 输出资源负载（load/capacity/utilization）并可视化（图/热力/行内叠加）（证据：B6/C2）  
18. 负载与甘特联动：点超载区可定位/高亮贡献任务（证据：C1/B6）  
19. 负载口径可配置：是否含换型/等待/停机等段（证据：B6/说明）

### 2.5 方案对比（你们的关键需求之一）
20. Scenario（基准/方案A/方案B）存储与切换（证据：B1/store 说明）  
21. 双面板或叠加对比（至少一种）+ 同步滚动/缩放（证据：C1/C2）  
22. Diff 高亮：开工/完工变化、资源变化、延期变化（证据：B6/C1）

## 3. 架构扩展性与可维护性检测清单（“能不能持续演进”）

> 评分同上：0/1/2；必须引用证据（A/B 类）。

### 3.1 分层边界（必须项）
1. Headless Core：核心计算不依赖 Vue/DOM（证据：A1/B2）  
2. ViewModel/布局计算为纯函数或可单测模块（证据：B2）  
3. Renderer 可替换：任务条与依赖线分别渲染（DOM + SVG 或其他）（证据：A1/B2）  
4. Interaction 独立：交互不直接改渲染层，不依赖 DOM 取值（证据：B3）  

### 3.2 命令模型与可回滚（强烈建议）
5. 交互以 Command 提交（execute/do/undo），支持 Undo/Redo（证据：B5/B3）  
6. Command 可序列化为 patch（便于保存/审计/协同/方案）（证据：B5）  

### 3.3 规则/校验可扩展（APS 规则永远加不完）
7. constraintEngine / ruleRegistry：规则可插拔而不是散落 if else（证据：B6）  
8. 校验结果结构化：`ok/severity/reason/autoFix?`（证据：B6）  

### 3.4 插件化与扩展点（长期维护关键）
9. Feature plugins：拖拽/连线/多选/菜单等按模块注册（证据：A1/B3）  
10. Registry：列/菜单/任务样式等可注册扩展（证据：A1/说明/代码）  
11. Adapter：后端 DTO 与前端 Domain 解耦（fromApi/toPatch）（证据：A1/B1）  

### 3.5 可测试性
12. core/layout/constraints 有单测或可被单测（证据：测试目录/说明）  
13. 关键交互有最小集成测或可复现脚本（证据：说明）

## 4. 常见“红旗”反模式（出现就要扣分）

AI 请明确指出是否存在以下问题，并给出证据位置：
1. **布局计算写在 Vue 组件里**，耦合视图生命周期（难测、难复用）  
2. 拖拽过程中直接 `task.start = ...` 改 store，无命令栈（难回滚、难审计）  
3. 校验逻辑散落在多个 handler 里（规则膨胀后维护灾难）  
4. 依赖线与任务条强耦合、无法按可视区/折叠隐藏（连线复杂后痛苦）  
5. Scenario/方案对比靠复制整份数据结构而非差量/映射（内存与复杂度上升）

## 6. 评分与结论等级（用于快速决策）

> 说明：总分 = 业务能力（满分 44）+ 架构能力（满分 26）= 70

- **A（APS 可用且易扩展）**：≥ 55 分，且第 3 节 1-4 项均 ≥1 分  
- **B（基本可用但需关键重构）**：45~54 分，或命令栈/规则引擎缺失  
- **C（展示型甘特为主）**：30~44 分，缺少日历/约束/批量/方案等多项  
- **D（不建议作为 APS 甘特基座）**：< 30 分，或核心逻辑高度耦合 UI、难以演进
